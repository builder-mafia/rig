# 코드 리뷰 스타일 가이드

## 요약

이 문서는 최근 `config-file` 리팩터링 과정에서 정리된 코드 리뷰 기준을 문서화한 것이다.
핵심 방향은 복잡한 상태를 명시적으로 표현하고, 데이터와 렌더링 책임을 가까운 곳에 두며, 불필요한 추상화 없이 단순한 구조를 유지하는 것이다.

## 핵심 원칙

1. 화면 상태는 명시적으로 표현한다.
2. 데이터 로딩은 실제로 사용하는 컴포넌트 가까이에 둔다.
3. Provider 는 공용 상태만 가진다.
4. 리스트 렌더링은 작은 컴포넌트로 나누되, 과한 분리는 피한다.
5. 같은 역할의 트리 항목은 하나의 컴포넌트로 재귀적으로 처리하는 쪽을 우선한다.
6. 기존 도메인 타입을 그대로 쓸 수 있으면 그대로 쓴다.
7. 메인 화면이 어떤 UI 를 보여줄지 상위에서 중앙집중적으로 결정한다.
8. React/Next.js 성능 관점에서는 Vercel Best Practices 를 기본 기준으로 삼는다.

## 화면 상태

- `showCreateForm` 같은 임시 boolean 으로 화면을 제어하지 않는다.
- 실제 의미가 화면 전환이라면 `pane`, `mode`, `view` 같은 명시적인 상태로 표현한다.
- 이 상태는 메인 워크벤치나 상위 Provider 가 소유한다.
- 하위 컴포넌트는 상태를 직접 해석해서 UI 를 켜고 끄기보다, 이미 결정된 상태를 받아 렌더링에 집중한다.

예시:

```ts
const [pane, setPane] = useState<'content' | 'create-entry'>('content');
```

## Provider 책임

- Provider 는 여러 컴포넌트가 함께 써야 하는 공용 상태만 가진다.
- 특정 리스트의 로딩 여부처럼 지역적인 UI 상태는 Provider 로 올리지 않는다.
- 상위에서 제어해야 하는 화면 상태는 Provider 에 둬도 되지만, 단지 버튼 하나를 누르기 위해 별도 전역 스토어를 만드는 것은 피한다.
- 가능하면 `use(ConfigFileWorkbenchContext)` 처럼 어떤 컨텍스트를 쓰는지 명시적으로 드러낸다.

좋은 예:

- 현재 선택된 파일
- 확장된 폴더 상태
- 편집 중인 값
- 현재 메인 패널이 어떤 pane 인지

나쁜 예:

- 특정 리스트 한 곳에서만 쓰는 `isLoadingList`
- 하위 컴포넌트 내부에서만 의미가 있는 임시 UI 토글

## 데이터 로딩

- 어떤 데이터가 한 컴포넌트에서만 필요하다면 그 컴포넌트 안에서 직접 가져온다.
- 리스트 데이터와 그 로딩 상태는 리스트 컴포넌트가 책임진다.
- 데이터를 가져오는 위치와 그 데이터를 렌더링하는 위치가 멀어질수록 구조가 흐려진다.
- React Query 는 유용한 선택지지만, 단순한 로딩이라면 더 작은 구조가 더 낫다.

기준:

- 공용 상태면 Provider
- 지역 상태면 해당 컴포넌트

## 컴포넌트 분리 기준

- 큰 `map()` 블록은 보통 별도 Item 컴포넌트로 분리한다.
- 파일과 디렉토리처럼 UI 형태가 명확히 다르면 분리한다.
- 그러나 루트용 컴포넌트, 중첩용 컴포넌트처럼 같은 역할을 이름만 다르게 둘로 나누는 것은 다시 검토한다.
- 분리의 목적은 재사용성과 가독성이지, 파일 개수를 늘리는 것이 아니다.

좋은 방향:

- `EntryListView`
- `EntryItemView`
- `FileItemView`
- `DirectoryItemView`

피해야 할 방향:

- `RootEntryItemView`, `NestedEntryItemView` 처럼 역할이 거의 같은 컴포넌트를 데이터 위치만 다르다는 이유로 따로 두는 것
- 작은 차이를 처리하려고 컴포넌트 계층을 과도하게 늘리는 것

## 트리 렌더링

- 트리 구조는 가능하면 `EntryItemView` 하나가 재귀적으로 처리한다.
- 루트 항목과 자식 항목이 본질적으로 같은 항목이라면 같은 컴포넌트로 처리하는 쪽이 더 단순하다.
- 디렉토리는 내부에서 자식들을 `map()` 하면서 다시 `EntryItemView` 를 호출하면 된다.
- 중첩 렌더링을 위해 별도의 전용 컴포넌트를 추가하기 전에, 기존 엔트리 컴포넌트를 그대로 재귀 호출할 수 있는지 먼저 본다.

좋은 예:

```tsx
{directoryEntries.map(child => (
  <EntryItemView
    key={child.path}
    rootEntry={rootEntry}
    entry={child}
    depth={depth + 1}
  />
))}
```

## 타입 사용 원칙

- 기존 도메인 타입이 충분하면 새 ViewModel 을 만들지 않는다.
- `StorageConfigFile` 이 이미 필요한 필드를 가지고 있다면 그대로 사용한다.
- 단순히 필드를 몇 개 옮기기 위한 변환 함수는 만들지 않는다.
- 타입을 좁히는 것이 구조를 단순하게 만드는지, 아니면 오히려 이해 비용만 늘리는지 먼저 본다.

## 렌더링 구조

- 어떤 pane 을 렌더링할지는 메인 화면이 결정한다.
- 하위 컴포넌트는 자신이 렌더링될지 말지를 판단하는 역할까지 떠안지 않는다.
- 조건부 렌더링은 여러 군데에 흩뿌리기보다 상위에서 한 번 결정하는 쪽이 좋다.

좋은 예:

```tsx
return pane === 'create-entry' ? (
  <ConfigFileCreateFormView />
) : (
  <ConfigFileContentView />
);
```

## 단순함 우선

- 맞는 구조가 두 개라면 더 단순한 쪽을 선택한다.
- 새로운 전역 상태, atom, manager, helper 를 만들기 전에 기존 Provider 나 기존 컴포넌트 안에서 해결 가능한지 먼저 본다.
- 한 단계의 추상화가 실제 문제를 해결하지 않으면 넣지 않는다.
- 코드 리뷰의 목표는 “더 추상적인 코드”가 아니라 “더 읽기 쉬운 코드”다.

질문 기준:

- 이 상태는 정말 여러 곳에서 공유되는가?
- 이 컴포넌트 분리는 역할이 다른가, 아니면 위치만 다른가?
- 이 helper 는 복잡함을 줄이는가, 아니면 이름만 추가하는가?
- 이 boolean 은 사실 더 풍부한 상태를 숨기고 있지 않은가?

## Vercel React Best Practices 반영 기준

React/Next.js 코드에서는 Vercel Best Practices 를 기본 성능 기준으로 반영한다.
다만 이 프로젝트에서는 성능 최적화보다 먼저 구조 단순화와 책임 분리가 맞는지 본다.

특히 자주 반영할 항목:

### 1. 불필요한 리렌더 줄이기

- 단순한 값 계산에 `useMemo` 를 남발하지 않는다.
- effect 안에서 파생 상태를 다시 세팅하지 말고 render 중 계산 가능한 값은 그대로 계산한다.
- 콜백에서만 쓰는 값을 읽기 위해 불필요하게 subscribe 하지 않는다.

### 2. 번들 크기 줄이기

- barrel import 보다 직접 import 를 선호한다.
- 무거운 컴포넌트는 필요할 때만 `dynamic()` 으로 불러온다.

### 3. 데이터 패칭 위치 개선

- fetch 는 가능한 한 실제 소비 지점 가까이 둔다.
- 상위에서 미리 다 가져와 하위로 무조건 전달하는 패턴은 신중하게 쓴다.

### 4. 조건부 렌더링 명확화

- 렌더링 분기는 상위에서 명확하게 드러나야 한다.
- 복잡한 `&&` 체인보다 의미가 분명한 ternary 나 명시적 분기를 선호한다.

## 리뷰 체크리스트

- 화면 상태가 boolean 남발 없이 명시적으로 표현되어 있는가?
- Provider 가 진짜 공용 상태만 들고 있는가?
- 데이터 fetch 책임이 너무 위로 올라가 있지 않은가?
- 큰 `map()` 블록이 적절한 Item 컴포넌트로 분리되어 있는가?
- 반대로 컴포넌트가 불필요하게 잘게 쪼개지지는 않았는가?
- 루트와 하위 엔트리를 같은 컴포넌트로 처리할 수 있는데 굳이 나누고 있지는 않은가?
- 타입 변환 helper 가 실제로 필요한가?
- 메인 화면이 렌더링 결정을 중앙에서 하고 있는가?
- Vercel React Best Practices 관점에서 불필요한 리렌더나 번들 낭비가 없는가?

## 결론

좋은 구조는 화려한 추상화보다 명확한 책임 분리에 가깝다.
상태는 의미 있게 표현하고, 데이터는 가까운 곳에서 가져오고, 같은 역할의 UI 는 같은 컴포넌트로 처리하며, 필요 이상으로 쪼개지 않는다.
이 문서의 목적은 코드를 “복잡하게 잘 나누는 것”이 아니라 “단순하게 오래 유지보수할 수 있게 만드는 것”이다.
