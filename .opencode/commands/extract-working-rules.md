## 사용자 입력

```text
$ARGUMENTS
```

당신은 방금 있었던 사용자-어시스턴트 간 구현 방향 불일치를 분석한다.

당신의 역할은 기능 요구사항을 다시 설명하는 것이 아니다.
당신의 역할은 구조적으로 무엇이 잘못되었는지 추출하고, 다음 작업부터 바로 적용할 수 있는 재사용 가능한 작업 규칙으로 정리하는 것이다.

## 핵심 목표

다음 구현이 아래 상태를 유지하도록 돕는 짧은 구조 회고 문서를 만든다.

- 구조적으로 단순할 것
- 레이어 경계가 분명할 것
- 재사용 가능할 것
- 이해하기 쉬울 것
- 하위 레이어에 불필요한 도메인 개념이 들어가지 않을 것

사용자는 기능을 나열하듯 확장하는 접근보다, 구조적으로 더 단순한 설계를 훨씬 중요하게 생각한다.

## 반드시 확인할 질문

분석할 때 항상 아래를 점검한다.

1. 상위 레이어의 도메인 개념이 하위 레이어로 내려갔는가?
2. 실제 책임보다 더 많은 타입, 함수, 단계가 생겼는가?
3. 여러 함수를 하나의 더 일반적인 primitive 로 줄일 수 있는가?
4. 동작은 일반적인데 이름만 특정 기능에 종속되어 있는가?
5. 표면적인 요구사항만 따라가고, 더 단순한 구조 경계를 놓쳤는가?
6. 무엇이 제품 로직이고, 무엇이 일반 시스템 로직인가?
7. lower layer 가 몰라도 되는 개념을 알고 있지는 않은가?

## 선호하는 설계 편향

항상 아래 원칙을 우선한다.

- 하위 레이어는 최대한 generic 하게 유지한다
- 제품 개념은 상위 레이어에 남긴다
- feature-specific helper 여러 개보다 general primitive 하나를 선호한다
- 실제 도메인 개념이 아니라면 새 엔티티를 추가하지 않는다
- 꼭 필요하지 않다면 UI/제품 용어를 Rust/backend/storage 레이어에 넣지 않는다
- 함수가 사실상 "로컬 경로가 존재하는지 확인"하는 역할이라면 이름도 일반적으로 짓는다
- 등록, 선택, preset, 추천 같은 제품 의미는 frontend/application 레이어에 둔다

## 출력 형식

결과는 반드시 한국어 Markdown 으로 작성하고, 아래 섹션을 정확히 포함한다.

### 1. 무엇이 문제였는가
- 2-4개 bullet
- 대화 태도 문제가 아니라 구조적 문제에만 집중한다

### 2. As-Is vs To-Be
- 반드시 표 또는 bullet 비교 형식으로 작성한다
- As-Is 에는 실제로 복잡해졌던 구조를 적는다
- To-Be 에는 더 단순하고 일반화된 구조를 적는다
- 레이어 책임, 함수 이름, 타입 개수, 개념 배치 차이가 드러나야 한다

### 3. 더 나은 구조 경계
- 2-5개 bullet
- 각 레이어가 무엇을 알고, 무엇을 몰라야 하는지 적는다

### 4. 재사용 가능한 작업 규칙
- 3-7개 bullet
- 다음부터 바로 적용할 수 있는 규칙 문장으로 쓴다
- 짧고 단정적으로 쓴다

### 5. 리팩터링 방향
- 2-5개 bullet
- 현재 구현을 어떤 방향으로 단순화해야 하는지 적는다

### 6. 다음 요청부터 따를 프롬프트
- 짧고 재사용 가능한 지시문 블록을 작성한다
- 다음 구현 요청에 그대로 붙여 넣을 수 있어야 한다
- 구체적이고 명령형으로 작성한다

## 중요 제약

- 사용자를 칭찬하지 않는다
- 장문의 에세이처럼 쓰지 않는다
- 말투나 배려 표현에 집중하지 않는다
- 자명한 구현 디테일을 길게 설명하지 않는다
- "소통을 더 잘하자" 같은 일반론을 쓰지 않는다
- 실제 다음 코드 구조에 영향을 주는 수준으로 구체적으로 쓴다
- 약한 제안보다 강하고 명확한 문장을 선호한다

## 좋은 방향 예시

좋음:

- Rust 는 로컬 경로 검사만 담당한다
- preset 메타데이터는 프론트에 둔다
- 기능별 path API 여러 개 대신 일반적인 local path inspection API 하나로 정리한다

나쁨:

- 구현할 때 더 신중하자
- 아키텍처를 더 깊게 생각하자
- 다음에는 커뮤니케이션을 더 잘하자

## 최종 기준
